#Linux

十个线程同时对count加十万，输出发现不能准确达到100万

原因：

count被分解为三条原子操作

在执行过程中发生了线程切换，导致访问临界资源时出现数据竞争

解决方法：

  • 互斥锁(mutex):

       pthread_mutex_lock(&mutex);
   (*pcount)++;
   pthread_mutex_unlock(&mutex);

    互斥锁访问的临界资源被占用时会让出CPU进入阻塞态。在临界资源占用时间较长时使用互斥锁，比如线程安全的rbtree

  • 自旋锁(spinlock):

    自旋锁访问的临界资源被占用时会保持等待，如果临界资源被占用的时间非常短，自旋锁可以节省线程切换的开销

  • 原子操作：用单条CPU指令实现

    int inc(int *value,int add){

    int old;

    __asm__ volatile(         //内联汇编语法
        "lock; xaddl %2, %1;"
        :"=a" (old)
        :"m" (*value),"a"(add)
        :"cc","memory"  

    );

     return old;
}

线程池

服务器在服务多个客户端时，使用多线程的方式

而服务器资源有限，不能无限新建线程，故使用线程池：

1.避免线程太多，使得内存耗尽

2.避免创建与销毁线程的代价

3.任务与执行分离

例如对于日志文件，磁盘操作比内存操作慢很多，所以在写日志时会引起线程的挂起

将写日志文件的任务交给线程池

银行的例子：

  • 办业务的人 -> 任务队列

  • 柜员 -> 执行队列

  • 公示牌 -> 管理组件(防止多个办业务的人在一个柜台办理，两个柜员同时为一个客户办理)

    • 锁

线程池的实现

任务队列

struct nTask{
    void (*task_function)(void *arg);
    void *user_data;  //用来做任务执行的参数

    struct nTask *prev;
    struct nTask *next;
};

执行队列

struct nWorker{
    pthread_t threadid;

    struct nWorker *prev;
    struct nWorker *next;
};

管理组件

typedef struct nManager{
    struct nTask *tasks;
    struct nWorker *workers;

    pthread_mutex_t mutex;
    pthread_cond_t cond;  //条件变量

}ThreadPool;
